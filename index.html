クライアントサイド制約 突破サンドボックス

HTML の maxlength、ファイルの accept、disabled、 required、min/max といった フロントエンドの制約はセキュリティ境界ではないことを実演します。 ブラウザは利用者の手元で動くので、利用者は何でも書き換えられます。

結論（先に）: クライアントサイドの検証は 「正しい利用者の操作ミスを減らす UX のため」のもの。 本当の検証・認可・サイズ制限は必ずサーバー側で行うこと。 このサイトは攻撃ツールではなく、その理由を体感するための教材です。

デモ一覧

  • 1. maxlength（最大文字数）を突破
  • 2. ファイルアップロード制限（accept / サイズ）を突破
  • 3. disabled / readonly を解除
  • 4. required / pattern / type 検証を突破
  • 5. hidden フィールド改ざん（価格改ざん）
  • 6. select の選択肢制限を突破（権限昇格）
  • 7. number の min / max を突破
  • 8. JavaScript バリデーションを無効化
  • 9. 禁止ワードフィルタ / textarea maxlength を突破

そもそも、なぜ突破できるのか

サーバーから見ると、リクエストは「どのブラウザから来たか」「どんな UI を経由したか」を区別できません。 攻撃者（あるいは普通の利用者）は次の手段でフロントの制約をすべて回避できます。

  • DevTools の Elements で属性を削除・編集（maxlength 削除、disabled 解除など）
  • DevTools の Console から JS で値や DOM を直接操作
  • JavaScript を無効化（クライアント検証ごと無効化）
  • ブラウザを使わず curl / スクリプトで直接 HTTP リクエストを送信
  • プロキシ（Burp/mitmproxy 等）で送信途中のリクエストを書き換え

各デモではページ内のボタンで上記と同等の操作を再現し、 「サーバーが実際に受け取る値」を黒いパネルに表示します。 画面上の制約に関係なく、その値がそのまま送られることが確認できます。

GitHub Pages 等の静的ホスティングには本物のサーバーがないため、 「送信」は同一ページ内のモックで再現しています。 実システムでは、この同じデータが本物のサーバーに届きます。

教育・防御目的のデモです。自分が権限を持たないシステムに対してこれらの手法を使わないでください。
